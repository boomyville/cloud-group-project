Progress Up-to-Date

Progress Summary:

Spartan Cloud Solutions (SCS) is pleased to present a progress update on the ongoing cloud migration project for our client, ABC Logistics. This report highlights the successful completion of key milestones, particularly the deployment of 20 virtual machines (VMs) in the Azure cloud environment. Leveraging Linux (Ubuntu 20.04) as the operating system, SCS has seamlessly transitioned 70% of ABC Logistics' critical data to Azure Storage, paving the way for enhanced operational efficiency and scalability. In addition, SCS has implemented Azure Database for MySQL to host relational data from several business-critical applications, further enhancing the scalability and performance of ABC Logistics' cloud infrastructure.

Deployment Overview:

               

Figure 1: Current ABC Logistics Virtual Networks (VNets)

SCS has orchestrated the deployment of 20 virtual machines within Azure, strategically organised to cater to ABC Logistics' operational needs across different regions. These VMs have been divided into two Virtual Networks (VNets), distinguishing between the Eastern Australia and Southeast Australia regions. Each VNet boasts two secure subnets, ensuring optimal performance and security. Notably, the subnet configuration comprises redundant server pairs for load balancing and resilience, alongside additional servers to support diverse business functions.

Rationale for VM Utilisation:

The decision to opt for virtual machines over Platform as a Service (PaaS) solutions stems from the proprietary nature of ABC Logistics' business software. These applications necessitate specialised server configurations, rendering VMs the preferred hosting environment to ensure compatibility and functionality. Moreover, considering the substantial investment already made in ABC Logistics' proprietary non-PaaS solution, transitioning to a PaaS model would unlikely yield immediate cost savings, underscoring the pragmatic approach adopted in this migration endeavour.

Installed Applications:

Within the deployed virtual machines, a suite of mission-critical applications has been installed to bolster ABC Logistics' operational capabilities. These include Enterprise Resource Planning (ERP), with two VMs allocated in each of the two regional VNets, alongside essential systems such as Transportation Management, Warehouse Management, and Supply Chain Management. Additionally, a dedicated VM has been provisioned to facilitate Microsoft Enterprise Identity (Entra ID) server operations, further fortifying ABC Logistics' digital infrastructure[a][b][c].

The successful deployment of 20 virtual machines marks a significant milestone in ABC Logistics' cloud migration, underscoring SCS's commitment to delivering tailored solutions that align with our client's unique requirements and business objectives. As we continue to advance through this transformative endeavour, SCS remains dedicated to facilitating ABC Logistics' transition to a robust and scalable cloud infrastructure that drives sustained growth and operational excellence.

Utilisation of Ansible for VM Provisioning and Software Orchestration:

Spartan Cloud Solutions (SCS) has embraced Infrastructure as Code (IaC) principles in the orchestration of virtual machine provisioning and software installation, leveraging Ansible as the primary automation tool. Ansible's robust capabilities enable the codification of infrastructure configuration, facilitating the seamless deployment of virtual machines (VMs) in the Azure environment while adhering to industry best practices and security standards. By treating infrastructure as code, SCS ensures consistency, repeatability, and scalability throughout the deployment process, empowering ABC Logistics with a resilient and agile cloud infrastructure.

The benefits of application of IaC for VM provisioning and software deployment include:

  • Consistency: IaC ensures that infrastructure configurations are consistent across environments, reducing the risk of configuration drift and minimising human error.
  • Scalability: By codifying infrastructure, IaC enables organisations to scale resources up or down quickly and efficiently to meet changing demands.
  • Repeatability: With IaC, infrastructure deployments become repeatable and predictable, allowing for reliable and reproducible deployments across different environments.
  • Version Control: IaC enables infrastructure configurations to be version-controlled, providing visibility into changes and facilitating collaboration among teams.
  • Speed and Efficiency: Automation through IaC accelerates the deployment process, reducing manual effort and streamlining operations for increased efficiency.
  • Cost Savings: By optimising resource utilisation and minimising manual intervention, IaC helps organisations achieve cost savings through improved resource management.

Rationale for Choosing Ansible[d]:

The decision to deploy Ansible as the automation tool of choice was made after careful consideration of alternative solutions such as Terraform, Chef, Puppet and Rudder. While each tool offers unique capabilities, Ansible emerged as the preferred option for several compelling reasons. Unlike Terraform, which specialises in infrastructure provisioning and lacks robust configuration management features, Ansible provides a comprehensive solution that seamlessly integrates both infrastructure provisioning and software orchestration. Additionally, compared to Chef and Rudder, Ansible's agentless architecture eliminates the need for additional software installations on target systems, simplifying deployment and reducing complexity. Ansible's declarative syntax and idempotent nature ensure consistent configuration management, mitigating the risk of configuration drift, a feature lacking in Chef. Moreover, Ansible's extensive library of pre-built modules and robust community support offer unparalleled flexibility and scalability, enabling rapid development and deployment of automation workflows tailored to ABC Logistics' unique requirements. By selecting Ansible, SCS ensures a holistic approach to infrastructure automation, delivering a solution that combines the best of provisioning and configuration management capabilities, thus optimising efficiency and reliability in ABC Logistics' cloud migration journey.

Deployment of Azure Database for MySQL:

The adoption of Azure Database for MySQL by ABC Logistics represents a strategic decision aimed at leveraging a scalable, managed database service to accommodate the relational data requirements of critical business applications. This choice offers numerous benefits tailored to ABC Logistics' specific needs and operational objectives.

Reasons and Benefits:

  1. Scalability and Performance: Azure Database for MySQL provides seamless scalability, allowing ABC Logistics to adapt to fluctuating workloads and increasing data volumes without compromising performance. This ensures consistent application responsiveness and user experience, critical for a logistics operation with dynamic demands.
  2. Managed Service: By opting for a managed database service, ABC Logistics offloads the burden of database administration tasks such as patching, backups, and maintenance to Azure, freeing up resources to focus on core business activities and innovation.
  3. High Availability and Disaster Recovery: Azure's robust infrastructure ensures high availability and disaster recovery capabilities for the MySQL database, minimising downtime and data loss risks. This enhances the reliability and resilience of ABC Logistics' data infrastructure, crucial for maintaining operational continuity.

Data Migration Process:

The migration of data from multiple applications into Azure Database for MySQL necessitated a systematic approach to ensure data integrity and consistency. SCS undertook the following steps:

  1. Data Profiling and Analysis: Comprehensive analysis of data sources, including various databases and Excel tables, was conducted to identify inconsistencies, data types, and dependencies.
  2. Schema Mapping and Transformation: SCS mapped the existing schemas from disparate sources to the MySQL database schema, ensuring compatibility and consistency. Data transformation processes were implemented to address any discrepancies and align data structures.
  3. Incremental Data Migration: Due to the complexity of data sources and potential data inconsistency challenges, SCS adopted an incremental data migration approach. This involved migrating data in manageable batches, allowing for iterative validation and troubleshooting.

Challenges and Mitigation:

Despite meticulous planning, data migration encountered challenges primarily related to data inconsistency arising from disparate sources. SCS addressed these challenges through rigorous data profiling, schema mapping, and transformation processes. Additionally, iterative validation and reconciliation efforts will need to be employed to identify and rectify discrepancies, ensuring data integrity in the Azure Database for MySQL environment.

Overall, the deployment of Azure Database for MySQL underscores ABC Logistics' commitment to harnessing innovative technologies to enhance operational efficiency and scalability, facilitated by SCS's expertise in database management and migration.

Virtual Networking: current architecture

As part of its ongoing cloud migration, ABC Logistics has a partially-functional virtual network (VNets) for its virtual machines (VMs). However, in its current state, this VNet is less efficient than it should be and the possibility for future expansion of the network is problematic.

There are three primary software applications which are currently run on 20 Ubuntu 20.04 VM web servers in two Azure VNets (as above, other applications are run as PaaS solutions in Azure). These are located in two different Azure regions, Eastern Australia and Southeast Australia, representing geo-redundancy. Each VNet contains three subnets for separation of web servers running the following three different logistics-based applications:

  1. A Warehouse Management System (WMS), in this case SAP (System Analysis Program Development) Extended Warehouse Management. This WMS software is used to manage warehouse operations, including inventory tracking, picking, packing, and shipping.
  2. The OptimoRoute Route Optimization Software (ROS). Route optimization software helps optimise delivery routes for efficiency and cost savings.
  3. A Document Management System (DMS). DMS software helps manage and organise documents, contracts, and other important digital assets. ABC Logistics is using Google Workspace for its DMS.

This solution is only appropriate for very small-scale deployments and any attempts to scale up using this framework are likely to be inefficient, error prone, and costly.

Virtual Networking: recommended implementations

To address the above-mentioned challenges  and to optimise its cloud infrastructure, ABC Logistics should consider setting up and using Azure Virtual WAN. Azure Virtual WAN provides a centralised hub-and-spoke architecture that can greatly simplify the management, security, and scalability of ABC Logistics' VMs. By transitioning to Azure Virtual WAN, ABC Logistics can benefit from the various advantages (discussed below), all of which can be configured via the Azure Portal and/or open-source Infrastructure as Code (IaC) applications such as Ansible (see above).

Centralised Management and Configuration

One of the primary benefits of Azure Virtual WAN is its ability to centralise the management and configuration of network settings. Instead of individually configuring, deploying, and maintaining each VNet and its connectivity individually, ABC Logistics can use Azure Virtual WAN to define and enforce network policies centrally. This allows for scalability and growth, as it significantly reduces the complexity and overhead associated with managing a large number of VMs spread across multiple VNets and regions.

In terms of connectivity between VNets, and between on-premises endpoints and VNets/VMs, Azure Virtual WAN offers a much improved experience than the current practice of configuring SSH tunnels from individual on-premises machines to VMs. The Virtual WAN will allow for easy site-to-site (S2S) and point-to-site(P2S) Virtual Private Network (VPN) connections from on-premises devices to VNets. These services are encrypted connections which traverse the public internet. For additional security and speed (and cost), there is also the possibility of implementing Azure ExpressRoute. This alternative to a VPN is a private and more permanent connection which is leased over Microsoft’s own network. It is faster and more private as it does not travel across the public internet. It is also more expensive and more permanent.

Improved Security

Azure Virtual WAN integrates seamlessly with Azure Firewall, providing a robust and scalable firewall solution for ABC Logistics' VMs. By using Azure Firewall in this way, ABC Logistics can create and manage firewall rules that apply to all traffic flowing through VNets, or subnets (see Figure 8 below) , rather than simply for individual VMs. These firewall rules can be based on application rules, network rules, and other criteria, providing granular control over all network traffic.

Additionally, with Azure Firewall, ABC Logistics can also implement advanced security policies such as those often applied with an Intrusion Detection System (IDS) or an Intrusion Protection System (IPS), for example: application-based filtering rules, threat intelligence, and logging and monitoring, all of which will provide the VNets with a high level of protection from a variety of cyber threats. Additionally, Azure Firewall can be deployed centrally and can scale automatically to handle increasing traffic volumes, making it a scalable and efficient solution for network security.

Optimised Traffic Routing and Load Balancing

Azure Virtual WAN enables ABC Logistics to optimise traffic routing and implement load balancing across its VMs. By leveraging Azure Virtual WAN's routing capabilities, ABC Logistics can ensure that traffic is routed efficiently between VNets and regions, reducing latency and improving overall network performance. Additionally, Azure Virtual WAN's built-in load balancing capabilities can help distribute traffic evenly across VMs, ensuring high availability and scalability for ABC Logistics' applications.

Scalability and Flexibility

As ABC Logistics continues to grow and scale its cloud infrastructure, Azure Virtual WAN provides the scalability and flexibility needed to support its evolving needs. Azure Virtual WAN can easily accommodate new VMs, VNets, and regions, allowing ABC Logistics to scale its network infrastructure without the need for significant reconfiguration or downtime.

Exploring Docker containers and Kubernetes

As part of this report, the Spartan Cloud Solutions have found some recommendations that will be beneficial in the operations of ABC logistics in the long run. One of these recommendations is the containerisation of applications as a substitute to running virtual machines to host applications. Docker is the mainstream tool that is perfect for managing containers and will vastly improve the way we develop and deploy applications.

Overview

Here is a brief overview of the benefits of using Docker containers

  • Streamlined development lifecycle
  • Development environments are standardised with containers
  • This gives us consistency between different operating systems and different devices.
  • This consistency is the backbone for the principle of continuous integration and continuous delivery (CI/CD)
  • It solves the problem of "it works on my computer, but it doesn't work on yours"
  • Cross platform
  • Docker can run on almost anything
  • Examples include:
  • Laptops
  • Desktop computers
  • The cloud
  • Single-board computers such as Raspberry Pis
  • A hacked PS3 or Wii running Linux
  • Network attached storage (NAS) devices
  • Smart home hubs
  • Smart TVs with Linux based OS
  • Point-of-sale (POS) systems
  • Digital signage players (electronic displays that show ads)
  • Resource efficient
  • Docker is lightweight and does not put a heavy drain on the host resources
  • This allows us to deploy many containers with minimal overhead and either reduce resource costs on the public cloud or reduce the cost of the physical hardware running the docker instances
  • Isolation and security
  • Each container is an isolated bubble with its own libraries, networking and configuration; unlike running a virtual machine with several applications running on it simultaneously
  • This means if an application on one container goes haywire, other containers on the same host will not be affected

The move from applications running on virtual machines to applications running in containers requires several steps to be made to our existing application. This process may be easy for applications that use mainstream libraries or frameworks but may require extra work if the application relies on specific hardware or GUI for user input. For example, the route optimisation software that is currently still running on an on-premises server uses a special API that gathers data from GPS modules placed in various transportation vehicles. This API may need to be rewritten such that it can run standalone in a container.

Integrating Docker

Detailed below would be a step-by-step guide in converting applications into docker compatible ones

  • Identify the applications dependencies and libraries that it uses
  • Map out application data to be stored in a separate entity (containers are stateless so when a container is stopped, all data is lost) such as Azure Blob Storage or Azure Container Storage
  • Generate a docker file which is a blueprint of how to build your application image including instructions to install necessary dependencies as well how to install the application itself

As part of this process of containerisation applications, developers may want to move from creating 'desktop-centric' applications to a web-based application. This will improve the portability of the application and expand the type of devices which can run the application. Web-based applications have the benefit of being able to be run on mobile devices (such as an iPad or smartphone or even TVs using the built-in web browser), cross-platform consistency (a Linux-based application will only run on a Linux-based device whereas a web-based application can run on any device running a web browser), faster deployment (no installation required) and the ability to scale up/down as required.

As an extension of containerisation, Kubernetes can provide us the ability to scale our containers to meet demand. Whilst Docker is useful in deploying our applications into containers, Azure Kubernetes Service can manage these containers and ensure their operation remains optimum.

The Kubernetes effect

Kubernetes is a platform-agnostic container orchestration tool responsible for allocating and scheduling containers and providing abstracted functionality such as internal networking and file storage. It can provide scalability by replicating pods (containers) to meet demands and wind them down when not needed. It also has the ability to monitor the health of all of these elements and adjust them as necessary to keep the overall service active.

Some benefits of using Azure Kubernetes Service

  • Ability to assign a private IP address to a container (can also use a private endpoint)
  • Provides more control and flexibility with how 'nodes' are managed
  • Provides a Kubernetes API so professionals with Kubernetes will feel right at home

The impact of using Azure Kubernetes Service

Scalability

Container instances can be provisioned to meet demand and deallocated when demand is low, preventing performance bottlenecks and user frustration when applications are slow and unresponsive

Availability

AKS can provision new containers when a node fails to ensure minimal downtime

Affordability

AKS deployed nodes are often cheaper than running scale set VMs

DevOps Efficiency

AKS (and the concept of containerisation) is useful in providing a consistent platform for software development and testing

Integration

AKS integrates with various other Azure services such as Azure Monitor

Cost Considerations and Assumptions

Virtual machines

  • The DS2 v2 SKU is the SKU used for currently provisioned VMs
  • Whilst the DS2 v2 VM SKU is a good general purpose compute unit, there are other configurations that may be better for different applications
  • Pay as you go pricing used

Azure Database for MySQL

  • Uses flexible server as single server is deprecated (retired 16th September 2024)
  • D2as v4 SKU is general purpose but other SKUs can be considered for burst transactions or memory-intensive workloads; however, the nature of the database transactions used by ABC logistics does not align with these workloads
  • Pay as you go pricing used

Public Static IP addresses

  • The current configuration has a public IP address attached to each VM
  • There is a very small saving by setting up an internet-facing load balancer and only attributing the load balancer a public IP address (the load balancer will direct traffic to non-public facing VMs and thus only 1 public IP address is needed)

                                               Figure 8: Azure Firewall implementation

  • Azure firewall is an expensive service but plays an important role in mitigating attacks on networks
  • Implementing Azure Firewall on a subnet level instead of a VNet level will increase costs substantially (since each VNet has 3 subnets) but will provide more customizability
  • If the network expands, consider a hub-and-spoke topology in protecting network resources where a central hub hosts shared services such as Azure Firewall and connects to spoke networks via load balancers. This allows for one Azure Firewall per hub-and-spoke network but allows for scalability at the cost of increasing complexity.

Azure Cost Management tool

Azure Cost Management is a tool that can help identify potential savings with Azure services, create budgets to control spending and analyse usage data. Listed below are some examples of some of the ways Azure Cost Management can save money:

  • Identify if over-provisioning of resources (such as virtual machines) is occurring
  • If virtual machines are underutilised, we can remove some of them to save costs
  • If virtual machines are overpowered for the task, we can downgrade the SKU to a cheaper configuration
  • Cost analysis can give businesses a prediction of expected costs which is important for:
  • Managing cash flow
  • Avoiding bill shock
  • Providing insight on the size of their technology footprint
  • Identify which applications or departments are causing cost blowouts
  • With the use of resource tags, we can track the cost of individual components of different services
  • The use of different subscriptions can help separate costs between different departments

Some other considerations

Here are some other suggestions in reducing costs

  • Full migration from virtual machines to containers
  • Containers are much more lightweight and much cheaper to run
  • A container with 2 vCPU and 7GB of memory will cost $118.66 whereas the equivalent VM will cost $113.70
  • However, the VM will require a managed disk which will cost significantly more than Azure Container Storage (estimates for Azure Container Storage are unavailable as the service is in preview as of 9th May 2024)
  • In addition, a container with 2 vCPU and 7GB of memory will have a lot more capacity to run applications than a virtual machine as a virtual machine will have to run an operating system on top of the application (the OS will gobble a lot of the CPU and RAM utilisation thus reduce the amount available for the application)
  • This process will require applications to be refactored to be able to built as Docker images (this process was detailed previously in the section on Docker)
  • Host databases on virtual machines instead of using Azure Database for MySQL
  • Azure Databases for MySQL are pretty much a virtual machine running a MySQL server
  • By removing the use of Azure Databases for MySQL and migrating them to an existing virtual machine, we can save money
  • There are some negatives with this approach:
  • Added complexity since Azure Database for MySQL is a managed service whereas running it on a virtual machine requires the user to manage the database
  • Database will be competing with any other hosted applications for computing resources
  • Moving applications hosted on virtual machines to Azure App Service
  • Azure App Service provides a managed solution that removes the management of the underlying infrastructure (virtual machines)
  • It is our understanding that virtual machines are currently provisioned to run legacy proprietary applications; however, many of these applications have more modern and mainstream alternatives available which can be run in containers or via Azure App Service
  • The migration from a proprietary application to a mainstream cloud-centric solution may require a lot of initial work to setup the transfer of data
  • This will also require employees to learn how to use the new application

Considering Documentation

Best practices for documentation

All organisations need good documentation; this allows members to easily gain a grasp of a certain configuration or process and reduces the amount of explaining team members need to do to other team members. Good documentation is the foundation for cooperation and integration between different teams and it begins by utilising some of the tools and practices that DevOps culture fosters.

Action

Rationale

Document regularly

Things change rapidly and humans forget; documenting keeps the user and others informed of changes. Think of it as note-taking but not just for oneself.

Use version control

Version control tracks changes over time and allows multiple users to contribute to a document and provides various features such as rollback (reverting to previous version), audit trail (track who makes what changes) and the ability to merge changes (for collaboration)

Use automation tools

Tools such as Terraform and Ansible automate manual 'clicking' and provide the same result every time. This allows infrastructure deployment to be consistent and since these tools are templates, these templates can be shared amongst other team members

Tools for documentation

Variable names

Variable names can be descriptive and choosing good variable names can help developers identify the purpose of the variable without explicitly explaining to the reader.

In-line comments

Developers should use comments to explain hard-to-understand concepts or special circumstances with their code. This allows other collaborators to be able to better understand the code.

Wiki pages

For processes that are not code-based, establishing a wiki page can allow team members to contribute explanations to various processes that happen in the workplace and can act as a library for future team members to draw upon. Wiki pages require a little bit of orchestration to organise information but are a lot more user-friendly than a whole bunch of pdf or word files. Cloud Spartan Solutions recommend wiki.js installed on an nginx server with a mySQL database for this purpose.

GitHub

Share code and code templates on GitHub and use the built in readme.md file as well as markdown script to provide documentation to code. GitHub comes with a whole bunch of collaboration tools to merge changes, create pull requests and provide a space to easily share resources.

Conclusion (Future steps)

In conclusion, ABC Logistics is poised to undergo a significant transformation in its IT infrastructure, driven by the transition from an on-premises framework to a hybrid cloud-based structure. This process has commenced, but has not yet been completed, and requires that the following steps are undertaken based on key recommendations from Cloud Spartans Tech. These are: complete database migration, transition to an Azure Virtual WAN (which includes integration with Azure Firewall as well as load balancing and traffic management), and containerisation of PaaS web applications.

There are still some hurdles with the migration process such as data migration (inconsistencies) as well as insecure firewall network security groups and firewall configurations[e][f]. There are some future considerations to consider as well such as migrating the use of virtual machines and managed databases to either a container-based system (which still offers us control) or a platform-as-a-service offering (less overhead management at the cost of flexibility) such as Azure App Service. In addition, Azure offers many services to enhance networking and performance such as Azure load balancing and Azure traffic manager.

The migration process is a big project and many changes will need to be documented. Training will be required to ensure all team members are able to utilise the strengths of the new platform as well as ensure daily operations happen as smoothly as possible. This is only the beginning and this initial migration process is only scratching the surface of the power of Microsoft Azure. In the future, we hope you will engage with our consultants again and explore the other areas of Azure that ABC Logistics can explore such as AI, data analytics, application insights and much more.
